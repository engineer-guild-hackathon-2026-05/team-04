# AI 活用ログ — Team 04

審査項目「AI 活用度」の根拠資料。Day1〜Day3 を通して、AI ツールをどう使ったかをカテゴリ別に記録する。
**最低 1 日 3 件以上**の記入を目安に。

## 記入ルール

- カテゴリ（ブレスト / ペルソナ / フロントエンド MVP / バックエンド / スライド / デモ動画 …）ごとにエントリを立てる
- 開発の節目ごとに追記する（5 分単位で書くより、機能/論点単位で書く）
- プロンプトは「実際に投げた文」を残す（要約は審査時に減点要因になりうる）
- 出力評価は **採用 / 一部採用 / 不採用** の 3 値で正直に
- 「採用しなかった理由」も書くと AI 活用の判断力アピールになる

## 全体振り返り

- **AI が一番効いた場面**：
  - **Codex / Claude Code によるコード実装と PR レビュー補助**。API route、UI component、Supabase migration、regression test の初稿を AI に作らせ、人間は仕様判断・採否判断・最終確認に集中した。PR でも code review agent 的に差分を見せ、バグ候補や見落としを先に洗い出すことで、レビュー時間を短縮できた
  - **HTML/CSS によるプレゼンモック作成**。スピード・コスパ・編集のしやすさのすべてで効いた
    - **Genspark などのスライド生成 AI を使うより圧倒的に速い**
    - 文言の微修正は **指示一つで反映**、あるいは **自分で直接コードをいじれば即時** に変更できる
    - スライドツールに縛られず、本番デモにそのまま使える品質まで一気に持っていけた
- **AI に頼らなかった場面とその理由**：
  - **壁打ち、特に「ニッチ × ニッチから連鎖を生み出す」という本当に新しい価値を作る部分**。AI は既存のものにすぐ付け加えようとしたり、既知の領域からアナロジーを探してくる傾向が強く、「面白さの創出」という今回の課題には適切でないと感じた
  - Day 1 のブレスト（2 ラウンドとも不採用）がまさに象徴的で、AI に投げると "繋ぐ＝マッチング" のような既存パターンに収束してしまい、お題の核を外す
- **次回 Hackathon で改善したい AI 活用**：
  - **リアルタイム議事録 / リアルタイムキャッチアップ機能**を試したい。今回は音声録音・Google Meet で議事録は取れていたものの、「セッション中に AI が現在地を要約して返してくれる」運用は使えていなかった
  - 壁打ちが停滞した瞬間に AI を挟むかどうかの判断も、もう少し踏み込んでよかった。今回は「今回のお題（面白さ創出）には合わない」「発散中にさらに余計な発散をされても困る」という理由で使わなかったが、リアルタイム議事録的なツールでこれまでの議論を圧縮して返してもらえば、停滞解除のトリガーとして機能した可能性がある
  - ただし「議論中に AI 要約を読みながら議論を続ける」のは人間側の認知負荷が大きく、運用設計が必要
